## 四、体验与交互优化（高优先级）

### 4.1 智能聚合与防打扰

**问题**：多个事件同时触发时，灵动岛可能频繁切换

**解决方案**：

```csharp
public sealed class EventAggregationPolicy
{
    // 同类事件聚合
    public static bool ShouldAggregate(IslandEvent e1, IslandEvent e2)
    {
        // 音量连续调节 → 合并为单次事件
        // Caps Lock 快速切换 → 仅显示最终状态
        return e1.Source == e2.Source && (e2.Timestamp - e1.Timestamp).TotalMilliseconds < 800;
    }

    // 优先级队列
    public static int GetPriority(IslandEvent evt)
    {
        return evt.Type switch
        {
            "notification" => 100,  // 最高优先级
            "media" => 80,
            "lockkey" => 60,
            "volume" => 40,
            "clock" => 20,
            _ => 50
        };
    }

    // 静默期合并
    public static bool IsInQuietPeriod(IslandEvent evt)
    {
        // 午夜 00:00-06:00 降低非关键事件优先级
        var hour = DateTime.Now.Hour;
        return hour is >= 0 and < 6 && evt.Type != "notification";
    }
}
```

**UI 表现**：
- 聚合事件显示计数徽章：`🔊 音量 x3`
- 优先级高的插队显示，低优先级的排队等待

## 五、性能与稳定性（高优先级）

### 5.1 内存与 CPU 优化

**当前问题**：媒体轮询（500ms 间隔）+ 歌词获取可能占用资源

**优化策略**：
```csharp
// 1. 智能轮询降频
if (!IsMediaPlaying() && !settings.AlwaysVisible)
{
    pollInterval = 2000;  // 降低到 2 秒
}

// 2. 歌词获取节流
if ((DateTime.UtcNow - _lastEnrichmentTime).TotalSeconds < 30)
    return;  // 30 秒内不重复请求

// 3. 图片缓存 LRU
_albumArtCache = new LRUCache<string, BitmapImage>(maxSize: 20);

// 4. 事件去重
if (evt.Signature == _lastEventSignature)
    return;  // 忽略重复事件
```

**监控指标**：
- 空闲时 CPU < 0.5%
- 内存占用 < 80MB（单实例）

### 5.2 崩溃恢复与容错

**增强点**：
```csharp
public sealed class CrashRecoveryService
```

## 九、已具备的优势

FluidBar 当前架构已非常优秀：

| 方面 | 现状 | 价值 |
|------|------|------|
| **事件总线** | `EventBus` + `IslandEvent` | 解耦监控器与 UI，易扩展 |
| **策略模式** | `MediaSnapshotSelectionPolicy` 等 | 灵活的业务规则 |
| **弹簧物理** | `SpringValue` 自定义实现 | 流畅的 iOS 式动画 |
| **防抖缓存** | 歌词、进程、专辑封面缓存 | 低资源占用 |
| **Record 类型** | `MediaSnapshot` 等不可变模型 | 线程安全，易调试 |
| **插件系统** | `IIslandPlugin` 接口 | 源码级扩展友好 |

**建议**：保持现有架构，仅在必要时引入新抽象。
